# نقشهٔ راه قابلیت‌ها

بررسی قابلیت‌های رایج پنل‌های دایرکت اینستاگرام، و اینکه هرکدام با **API رسمی متا** چقدر شدنی است.

معیار ثابت ما: هر چیزی که endpoint رسمی نداشته باشد ساخته نمی‌شود — چون ریسکش محدودشدن اپ و از کار افتادن هم‌زمان همهٔ پیج‌های مشتریان است.

---

## وضعیت فعلی

| قابلیت | وضعیت |
| --- | --- |
| پاسخ خودکار دایرکت با کلیدواژه | ✅ ساخته شده |
| پاسخ خودکار کامنت (دایرکت + پاسخ عمومی تصادفی) | ✅ ساخته شده |
| پاسخ چندگامی (متن، مکث، ارجاع به اپراتور) | ✅ ساخته شده |
| ارسال عکس / صدا / فیلم با آپلود | ✅ ساخته شده |
| دکمه‌های پاسخ سریع | ✅ ساخته شده |
| محدودکردن قانون به یک پست | ✅ ساخته شده |
| فاصلهٔ زمانی per-contact | ✅ ساخته شده |
| صندوق پیام، مخاطبان، گزارش‌ها | ✅ ساخته شده |

## اولویت‌بندی پیشنهادی

### ۱. ویترین‌ساز (کارت‌های محصول)

**شدنی است.** متا این را با نام Generic Template ارائه می‌دهد:

- حداکثر **۱۰ کارت** در یک پیام
- هر کارت: تیتر (۸۰ کاراکتر)، زیرعنوان (۸۰ کاراکتر)، عکس، و حداکثر **۳ دکمه**
- نوع دکمه: `web_url` (لینک خارجی) یا `postback` (ادامهٔ سناریو داخل دایرکت)

**چرا اول:** بیشترین اثر روی فروش، و زیرساختش (آپلود عکس، دکمه‌ها) از قبل آماده است.

**کار لازم:** نوع گام جدید `SEND_CARDS`، سازندهٔ کارت در پنل، متد جدید در provider.

---

### ۲. آیس‌بریکر (چهار دکمهٔ ابتدای گفتگو)

**شدنی است.** API رسمی Ice Breakers دارد و مستقل از اتوماسیون‌ها روی خود پیج تنظیم می‌شود.

**کار لازم:** صفحهٔ تنظیمات per-account برای تعریف تا ۴ گزینه و اتصال هرکدام به یک کلیدواژه.

**نکته:** چون کاربر با زدن آیس‌بریکر عملاً همان متن را می‌فرستد، با موتور فعلی ما بدون تغییر کار می‌کند.

---

### ۳. فالو اجباری

**شدنی است.** فیلد `is_user_follow_business` در User Profile API وجود دارد و می‌گوید آیا این مخاطب پیج را دنبال می‌کند یا نه.

**کار لازم:** یک شرط در موتور: «اگر فالو نیست، پیام درخواست فالو + دکمهٔ «فالو کردم» بفرست؛ در غیر این صورت سناریوی اصلی».

**محدودیت واقعی:** مقدار این فیلد لحظه‌ای است. اگر کاربر بلافاصله آنفالو کند سیستم متوجه نمی‌شود — این ذاتی API است، نه ضعف پیاده‌سازی.

---

### ۴. پشتیبان هوشمند (پیگیری زمان‌دار)

**شدنی است** و زیرساخت صف (BullMQ) از تأخیر پشتیبانی می‌کند.

**محدودیت تعیین‌کننده:** پنجرهٔ ۲۴ ساعتهٔ پیام‌رسانی متا. بعد از آخرین پیام کاربر، فقط ۲۴ ساعت اجازهٔ ارسال داریم. به همین دلیل رقبا هم سقف را حدود ۲۲ ساعت گذاشته‌اند.

**کار لازم:** نوع گام `SCHEDULE_FOLLOWUP` با تأخیر، و لغو خودکار اگر کاربر در این فاصله جواب داد.

---

### ۵. فرم‌ساز چندمرحله‌ای

**شدنی است ولی بزرگ‌ترین تکه.**

موتور فعلی «حافظه» ندارد: هر پیام مستقل ارزیابی می‌شود. برای فرم باید بدانیم کاربر وسط کدام فرم و کدام سؤال است.

**کار لازم:** جدول وضعیت گفتگو، تعریف فرم و سؤالات، اعتبارسنجی پاسخ‌ها (تلفن، عدد، چندگزینه‌ای)، کلمات خروج، پیام پایانی، و خروجی اکسل.

**پیشنهاد:** بعد از چهار مورد بالا، به‌عنوان یک پروژهٔ مستقل.

## چیزهایی که نمی‌سازیم

| قابلیت | دلیل |
| --- | --- |
| **لایک خودکار کامنت** | متا هیچ endpoint یا فیلد نوشتنی برای لایک ندارد. `like_count` فقط خواندنی است و تنها پارامتر قابل ارسال روی کامنت `hide` است. هر پنلی که این را ارائه می‌دهد از روش غیررسمی استفاده می‌کند. |
| **پیامک** | خارج از اکوسیستم متاست و به سرویس پیامکی جداگانه نیاز دارد. در تعریف اولیهٔ پروژه کنار گذاشته شد. |
| **خرید و پرداخت درون‌پنل** | در تعریف اولیهٔ پروژه کنار گذاشته شد. |

**جایگزین ارزشمند برای لایک:** قابلیت `hide` رسمی است و می‌شود با آن «مخفی‌کردن خودکار کامنت‌های اسپم یا توهین‌آمیز» ساخت — برای پیج‌های پرترافیک ارزش واقعی دارد.

## نکتهٔ فنی که باید بررسی شود

مستندات رقیب می‌گوید در مسیر **کامنت → دایرکت** فقط یک نوع محتوا قابل ارسال است (فقط متن، یا فقط عکس، یا فقط کارت) و ترکیبشان ممکن نیست.

پیاده‌سازی فعلی ما برای هر گام یک پیام جدا می‌فرستد. باید در عمل آزمایش شود که آیا متا برای یک کامنت بیش از یک پاسخ خصوصی می‌پذیرد یا نه. اگر نپذیرد، الگوی درست همان است که رقیب پیشنهاد داده:

> کامنت → یک پیام با دکمه → کاربر دکمه را می‌زند → از اینجا به بعد گفتگوی عادی است و هر تعداد پیام مجاز است.

---

## پنجرهٔ ۲۴ ساعته — قاعده‌ای که بر همه‌چیز حاکم است

- پاسخ به کامنت از کسی که قبلاً پیام نداده: **یک پیام**، تا ۷ روز بعد از کامنت
- بعد از اینکه کاربر پاسخ داد: پنجرهٔ ۲۴ ساعته باز می‌شود و گفتگو آزاد است

به همین دلیل در هر سناریو، اولین پیام باید کاربر را به پاسخ‌دادن تشویق کند (دکمه، سؤال، یا دعوت به گفتگو). این یک ترفند بازاریابی نیست؛ شرط فنی ادامهٔ گفتگوست.
